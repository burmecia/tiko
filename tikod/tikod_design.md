# tikod — Tiko Control Plane Design

`tikod` is the control plane for Tiko. It manages the lifecycle of projects and branches,
orchestrates PITR recovery, enforces retention, and runs chunk GC. It is a standalone Rust
binary that talks to the same `SimStore` (and eventually real S3) that `s3worker` and
`s3smgr` use, but has no dependency on a running PostgreSQL process.

## Boundary with s3worker / s3smgr

| Concern | Owner |
|---|---|
| Chunk I/O (read/write blocks) | `s3smgr` / `s3worker` |
| Delta manifest upload at checkpoint | `s3smgr/checkpoint.rs` |
| pg_state archive upload at checkpoint | `s3smgr/checkpoint.rs` |
| Periodic base manifest materialization (live DB) | `s3worker/pitr_task.rs` |
| WAL archival (`archive_command` binary) | `tiko_archive` binary |
| WAL restore (`restore_command` binary) | `tiko_restore` binary |
| **Everything below** | **`tikod`** |

## Store Layout (reference)

```
Standard bucket
  {org}/metadata/org.json                     ← OrgMeta (org_id, created_at)
  {org}/metadata/{proj}/project.json          ← ProjectMeta (branch lineage, status,
                                                 current_server_id, assigned_server_id)
  {org}/metadata/{proj}/lease.json            ← ProjectLease (server_id, acquired_at,
                                                 expires_at, generation)
  {org}/pitr/{proj}/bases/{tl:08X}/{lsn}/manifest.bin  ← materialized base manifests (per-timeline)
  {org}/pitr/{proj}/deltas/{tl:08X}/{lsn}/manifest.bin ← per-checkpoint delta manifests (per-timeline)
  {org}/pitr/{proj}/deltas/{tl:08X}/{lsn}/pg_state.tar.zst ← pg_control, pg_xact, pg_multixact, pg_subtrans, pg_filenode.map
  {org}/pitr/{proj}/wal/{tl:08X}/{segment}    ← archived WAL segments
  {org}/chunks/{branch_id}/{chunk_path}/{tl:08X}/{lsn} ← versioned chunk objects (per-timeline)

Express bucket
  {org}/{proj}/chunks/{chunk_path}/latest     ← hot cache (latest version, ephemeral — local to hosting server)
  {org}/{proj}/nblocks/{rel_fork}             ← relation size hint

$PGDATA/tiko/eviction_log                     ← dirty chunk log (local file, not in store)
```

**Note on express cache**: hot cache objects are written to the hosting server's local
`SimStore` (or local disk in production). They are **ephemeral and non-authoritative** —
the standard bucket is always the source of truth. When a project migrates to a new server
the cache is abandoned on the old server and rebuilt on demand on the new one. This is safe
because WAL and the standard-bucket chunks guarantee correctness.

**Note on PITR recovery**: after recovering a project to `target_lsn`, `post_recovery_cleanup`
deletes all express `latest` objects (stale old-timeline data), all bases/deltas with
LSN > `target_lsn`, and all WAL under the old timeline prefix (bulk-delete using
`current_timeline_id` from `project.json`). The merged manifest at `target_lsn` is uploaded
as the new initial base and `current_timeline_id` is updated to the new timeline.
Versioned chunk objects at LSN > `target_lsn` are left for the next GC run.

## Modules

### 1. Org Lifecycle (`tikod/src/org.rs`)

An org is the top-level namespace. All store keys are rooted at `{org}/`. Each org
has a metadata file at `{org}/metadata/org.json`.

**Create org**
- `org_id` is provided by the caller (assigned by an external process, not generated by tikod)
- Write `{org}/metadata/org.json` with `{ "org_id": <id>, "created_at": <unix_ts> }`

**Delete org**
- Assert no projects remain (only `{org}/metadata/org.json` exists under `{org}/metadata/`),
  or pass `--force` to delete all projects first
- With `--force`:
  1. List all projects under `{org}/metadata/` (skip `org.json`) and delete each branch
     in dependency order (children before parents)
  2. Delete `{org}/metadata/org.json`
  3. Bulk-delete everything remaining under `{org}/` in both buckets
  4. Schedule chunk GC for all `{org}/chunks/` objects (or delete immediately if `--force`)
- Without `--force`: refuse if any `project.json` entries exist

### 2. Branch & Project Lifecycle (`tikod/src/lifecycle.rs`)

Primitives already exist in `s3worker/src/project.rs` (`create_branch`, `delete_branch`).
tikod wraps them with the surrounding steps that require process control.

**Create root project**
- Write `project.json` with no parent fields
- Nothing else — `initdb` inside the PG process will produce the first delta manifest

**Create branch**
1. Connect to the parent PostgreSQL and run:
   ```sql
   CHECKPOINT;
   SELECT pg_create_restore_point('<branch_name>');
   ```
   `CHECKPOINT` flushes dirty chunks and uploads the delta manifest; `pg_create_restore_point`
   writes a named WAL record that marks the exact branch LSN. The parent keeps running.
2. Call `s3worker::project::create_branch(sim, parent_ns, child_ns, branch_lsn)`
   - Writes `project.json` for child
   - Builds initial manifest from parent base + deltas up to `branch_lsn`
3. Set env vars for the child (`TIKO_ORG_ID`, `TIKO_PROJECT_ID`, `TIKO_BRANCH_ID`,
   `TIKO_PARENT_PROJECT_ID`, `TIKO_PARENT_BRANCH_ID`, `TIKO_BRANCH_CHECKPOINT_LSN`,
   `TIKO_ROOT_PATH`) and run `initdb` to create an empty PGDATA — `s3smgr` bootstraps
   an empty manifest from env vars (`ProjectCtx::bootstrap`) when `project.json` is absent
4. Overwrite `$PGDATA/global/pg_control`, `pg_xact/`, `pg_multixact/`, `pg_subtrans/`,
   `global/pg_filenode.map` by extracting the parent's `pg_state.tar.zst` at `branch_lsn`
5. Write `$PGDATA/tiko/recovery_manifest.bin` (point-in-time manifest at `branch_lsn`)
   and configure `restore_command` + `recovery.signal` — the branch starts via the
   standard recovery path, replaying WAL from `branch_lsn` forward (WAL archive fallback
   to parent via `tiko_restore`)

**Delete branch**
1. Stop PostgreSQL for the branch (if running); release its lease
2. Call `s3worker::project::delete_branch(sim, branch_ns)`
   - Removes express, PITR, metadata objects
   - Leaves chunk objects for GC
3. Schedule a GC run for `{org}/chunks/{branch_id}/`

**List projects / branches**
- Enumerate `{org}/metadata/` prefix in the standard bucket
- Return `Vec<ProjectMeta>`

### 3. PITR Recovery Preparation (`tikod/src/pitr.rs`)

Given a target LSN, prepare PGDATA for PostgreSQL point-in-time recovery.

The target is always an LSN. The GUI may display checkpoint timestamps to the user
and let them pick a point in time, but it resolves the selection to the corresponding
delta manifest LSN before calling tikod. tikod only operates on LSNs.

**Step 1 — Validate target LSN**
- Confirm `{org}/pitr/{proj}/deltas/{current_tl:08X}/{target_lsn}/manifest.bin` exists
  (where `current_tl` = `project_meta.current_timeline_id`)
- The target LSN must correspond to an exact delta manifest (i.e. a checkpoint boundary)

**Step 2 — Build point-in-time chunk manifest**
- Load base manifest with LSN ≤ `target_lsn`
- Apply all deltas with LSN ≤ `target_lsn` (same logic as `materialize_base`)
- Write the merged manifest to `$PGDATA/tiko/recovery_manifest.bin`
  (msgpack+zstd wire format, read by `s3worker::recovery::load_recovery_manifest`)

**Step 3 — Restore pg_state**
- Download `{org}/pitr/{proj}/deltas/{current_tl:08X}/{target_lsn}/pg_state.tar.zst`
- Extract into `$PGDATA`: `pg_control`, `pg_xact/`, `pg_multixact/`, `pg_subtrans/`,
  `pg_filenode.map`
- This gives PostgreSQL a consistent pg state at the target checkpoint

**Step 4 — Configure recovery**
- Append to `postgresql.conf`:
  ```ini
  restore_command = '/path/to/tiko_restore %f %p'
  recovery_target_lsn = '<target_lsn>'
  recovery_target_action = 'shutdown'
  ```
- `touch $PGDATA/recovery.signal`

  `recovery_target_action = 'shutdown'` is preferred over `'promote'` for tikod's use case:
  PG shuts itself down cleanly after reaching `target_lsn`. s3worker (a bgworker) exits with
  PG. tikod detects the process exit — no polling of `recovery.signal` removal needed, and
  no `clear_recovery_mode()` IPC to a running s3worker. PG cannot accept writes before tikod
  finishes the post-recovery cleanup.

  Note: recovery is required even though pg_state is already at `target_lsn`.
  The `pg_control` archived during a live checkpoint has `state = DB_IN_PRODUCTION`,
  so PostgreSQL must enter crash recovery. With `recovery_target_lsn = target_lsn` it
  immediately hits the barrier (no WAL is actually replayed) and shuts down.
  The chunk data and pg_state are already consistent at `target_lsn`; the recovery
  pass is structurally required but effectively a no-op.

**Step 5 — Return**
- Caller (or the next module) starts PostgreSQL in recovery mode; it shuts itself down
  once it reaches `target_lsn`; tikod detects the process exit and runs post-recovery cleanup

**Post-shutdown cleanup** (performed by `orchestrate::post_recovery_cleanup` after
detecting process exit; applies to **both** new branch/project provisioning and PITR restore):
- Upload `$PGDATA/tiko/recovery_manifest.bin` to
  `{org}/pitr/{proj}/bases/{new_tl:08X}/{target_lsn}/manifest.bin` in the standard bucket —
  (uses the new timeline read from pg_control after shutdown)
  it is already the fully-merged manifest at `target_lsn`, so it becomes the initial
  base manifest for this project with no extra merging needed, regardless of whether
  this was a PITR restore or a new branch provisioned from a parent checkpoint
- Delete `$PGDATA/tiko/recovery_manifest.bin` and `$PGDATA/tiko/recovery_manifest_local.bin`
  from disk (if left, the next s3worker startup would reload them and re-enter recovery mode)
- Remove `recovery_target_lsn` and `recovery_target_action` lines from `postgresql.conf`
  and delete `$PGDATA/recovery.signal`
- Start PostgreSQL normally — s3worker starts fresh without recovery mode

**List available recovery points**
- Return all delta manifest LSNs within the retention window for a given project,
  along with their `checkpoint_timestamp` — the GUI uses these timestamps to build
  the point-in-time picker, then passes the chosen LSN back to tikod

### 4. Recovery Orchestration (`tikod/src/orchestrate.rs`)

The same recovery pass is used for two different callers:

| Caller | Has running PG? | PGDATA source |
|---|---|---|
| PITR restore (§3) | Yes — must stop first | Existing PGDATA of the project |
| New branch/project (§2) | No | Fresh PGDATA from `initdb` |

```
# ── PITR restore path ─────────────────────────────────────────────────────────
stop_postgres(pgdata)                 # only for PITR; skip for new branch/project
  └─ compute_backend.stop()

prepare_recovery(sim, ns, pgdata, target_lsn)   ← calls pitr::prepare_recovery()
  # writes recovery_manifest.bin, restores pg_state, configures postgresql.conf

# ── Shared path (PITR restore + new branch/project) ──────────────────────────
start_postgres_recovery(pgdata)
  └─ compute_backend.start()   # PG runs with recovery_target_action = 'shutdown'

wait_for_recovery_shutdown(pgdata, timeout)
  └─ wait for PG process exit (recovery reaches target_lsn → clean shutdown)

post_recovery_cleanup(sim, ns, pgdata, target_lsn)
  # ── Store cleanup (old timeline) ──────────────────────────────────────────
  old_tl = project_meta.current_timeline_id   # read from project.json before recovery
  └─ delete all delta manifests with LSN > target_lsn from {org}/pitr/{proj}/deltas/{old_tl:08X}/
     (old timeline — no longer reachable; list_restore_points must not offer them)
  └─ delete all base manifests with LSN > target_lsn from {org}/pitr/{proj}/bases/{old_tl:08X}/
     (superseded; new base is the one we're about to upload)
  └─ bulk-delete {org}/pitr/{proj}/wal/{old_tl:08X}/   # prefix-delete entire old timeline WAL
     (timeline ID changes after recovery; old WAL is unreachable via restore_command)
  new_tl = read_timeline_from_pg_control(pgdata)   # PG wrote new TLI at shutdown
  └─ update project.json: current_timeline_id = new_tl
  # NOTE: versioned chunk objects at LSN > target_lsn in standard bucket are
  # orphaned but left for the next GC run to collect via manifest cross-check.

  # ── Express cache ──────────────────────────────────────────────────────────
  └─ delete all {org}/{proj}/chunks/*/latest from the express bucket
     (stale data from the old timeline; if served, s3worker would return incorrect
      blocks on normal startup before the first checkpoint overwrites them)

  # ── New base manifest ──────────────────────────────────────────────────────
  └─ upload recovery_manifest.bin → standard bucket as bases/{new_tl:08X}/{target_lsn}/manifest.bin

  # ── Local files ────────────────────────────────────────────────────────────
  └─ delete recovery_manifest.bin + recovery_manifest_local.bin
  └─ remove recovery_target_lsn / recovery_target_action from postgresql.conf
  └─ delete recovery.signal

start_postgres(pgdata)
  └─ compute_backend.start()   # PG starts normally; s3worker has no recovery_manifest.bin

mark_active(sim, ns)
  └─ update ProjectMeta.status = "active"
```

### 5. Retention Enforcement / GC (`tikod/src/gc.rs`)

Called on a schedule (e.g., daily) per org. Referenced in `pitr_task.rs` as
`enforce_retention_org`.

In a multi-server cluster, GC for a given org must be held by exactly one server at a
time. Before running, acquire the GC lease at `{org}/gc_lease.json`
`{ server_id, expires_at }`. Skip silently if another server holds it.

**Retention policy: checkpoint-count-based, not time-based.**
A time-based cutoff would incorrectly delete data from inactive projects — a project
paused for weeks or months still has a valid current state. Counting checkpoints ties
retention to database activity: a busy project accumulates checkpoints quickly; an idle
one keeps all its history indefinitely until it generates enough new checkpoints to
trigger cleanup.

Policy: keep the last `max_checkpoints` (default 500) delta manifests per project.
Projects with fewer than `max_checkpoints` deltas are never pruned regardless of wall-clock time.

**Delta manifest GC**
- List all delta LSNs across all timelines under `{org}/pitr/{proj}/deltas/`, sorted ascending
  (GC uses the flat prefix to scan cross-timeline; `delta_prefix()` not `delta_prefix_for_timeline()`)
- If `total > max_checkpoints`: `cutoff_lsn = delta_lsns[total - max_checkpoints]`
  (i.e. the oldest LSN to keep is at position `total - max_checkpoints`)
- Delete all delta manifests (and their `pg_state.tar.zst`) with LSN < `cutoff_lsn`
- Projects with ≤ `max_checkpoints` deltas: skip, nothing to delete

**Base manifest GC**
- Keep all base manifests with `base_lsn ≥ cutoff_lsn` across all timelines
- Also keep the newest base with `base_lsn < cutoff_lsn` (serves as the recovery anchor
  for the oldest retained delta)
- Delete all older bases

**WAL GC**
- WAL segments are only needed to replay from the oldest retained checkpoint to the
  target recovery point; the oldest retained delta LSN is `cutoff_lsn`
- Delete WAL segments whose end LSN is entirely before `cutoff_lsn`
- Projects with ≤ `max_checkpoints` deltas: no WAL GC

**Chunk GC**
- After any branch deletion or after delta GC, versioned chunk objects may be orphaned
- Enumerate all live base manifests + retained deltas across the org to build the
  set of referenced `(branch_id, chunk_path, timeline_id, lsn_hex)` keys
- Delete any versioned chunk objects not in that set
- Zero branch (`branch_id = 0`) is permanent — never collected
- This is the most expensive operation; run it at low priority / off-peak

Thin wrapper around `s3worker::pitr_task::materialize_base` for projects whose
`s3worker` is not currently running (stopped instances, just-restored branches).

- Call `materialize_base(sim, ns)` synchronously
- Used before `prepare_recovery` to ensure the base manifest is up to date

### 7. WAL Archive Health (`tikod/src/wal_health.rs`)

In daemon mode, only check projects for which this server holds a lease.

- Read `ProjectMeta.last_archived_lsn` (to be added to `ProjectMeta`)
- Compare against the current insert LSN (obtained from `pg_current_wal_lsn()` via
  `psql -c` or a direct PG connection)
- Expose lag as a metric / status field
- Alert if no WAL segment has been archived within `archive_timeout * 2`

### 8. CLI & Daemon Surface (`tikod/src/main.rs`, `tikod/src/api.rs`)

tikod has two execution modes:

**One-shot CLI** — invoke a single operation and exit. All core logic lives in the
module functions; `main.rs` just parses arguments and calls them. This is the primary
way to drive tikod from external orchestration scripts or cron jobs.

**Daemon mode** (`tikod serve`) — long-lived process that exposes the HTTP API and
runs background tasks (periodic GC, WAL health polling). The HTTP handlers are thin
wrappers over the same module functions used by the CLI. GC and WAL health run on a
Tokio interval inside the daemon's runtime.

```
              ┌─ CLI mode ──────────────────────────────────────────┐
              │  tikod <subcommand> [args]                          │
              │  calls core fn, prints result, exits                │
              └────────────────────────────────────────────────────-┘
                                     │
                              core module fns
                    (org.rs / lifecycle.rs / pitr.rs / gc.rs …)
                                     │
              ┌─ Daemon mode ────────────────────────────────────────┐
              │  tikod serve [--addr 0.0.0.0:8080]                  │
              │  ├─ HTTP API  (axum)  ← thin wrappers               │
              │  └─ background tasks (tokio::interval)              │
              │       ├─ cluster::heartbeat_loop   (every 10s)      │
              │       ├─ cluster::failover_loop    (every 30s)      │
              │       ├─ lease::renew_held_leases  (every 15s)      │
              │       ├─ gc::enforce_retention_org (daily)          │
              │       └─ wal_health::check_lag     (every 60s)      │
              └─────────────────────────────────────────────────────┘
```

**CLI subcommands**

```
tikod create-org      --org <id>
tikod delete-org      --org <id> [--force]

tikod create-project  --org <id> --project <id> --branch <id>
tikod create-branch   --org <id> --project <id> --branch <id> \
                      --parent-project <id> --parent-branch <id> --lsn <hex>
tikod delete-branch   --org <id> --project <id> --branch <id>
tikod list-projects   --org <id>

tikod prepare-recovery --org <id> --project <id> --branch <id> \
                       --target-lsn <hex/hex> --pgdata <path>
tikod restore          --org <id> --project <id> --branch <id> \
                       --target-lsn <hex/hex> --pgdata <path>
                       # full cycle: prepare + start + wait-for-promotion

tikod list-restore-points --org <id> --project <id> --branch <id>

tikod gc               --org <id> [--max-checkpoints <n>]
tikod materialize      --org <id> --project <id> --branch <id>

tikod serve            [--addr <host:port>]   # daemon mode
```

### 9. Compute Backend Abstraction (`tikod/src/compute.rs`)

tikod must control the compute process hosting PostgreSQL. Currently this is a local
process (`pg_ctl`), but the Firecracker microVM target requires VM lifecycle control.

```rust
trait ComputeBackend {
    /// Start the compute unit (process or VM) with the given PGDATA and env.
    fn start(&self, pgdata: &Path, env: &[(String, String)]) -> Result<()>;
    /// Stop the compute unit gracefully.
    fn stop(&self, pgdata: &Path) -> Result<()>;
    /// Returns true if the compute unit is currently running.
    fn is_running(&self) -> bool;
    /// Issue a checkpoint and wait for it to complete.
    fn checkpoint(&self) -> Result<()>;
}
```

**`LocalProcess`** (current implementation)
- `start` → `pg_ctl start -D $PGDATA`
- `stop`  → `pg_ctl stop -D $PGDATA`
- `checkpoint` → `psql -c CHECKPOINT`

**`FirecrackerVm`** (future)
- `start` → Firecracker API: create VM, attach rootfs/block device containing PGDATA,
  boot with tiko env vars injected via kernel cmdline or vsock
- `stop`  → Firecracker API: send ACPI shutdown or `SendCtrlAltDel`, wait for VM exit
- `checkpoint` → vsock or TCP connection to postgres inside VM → `CHECKPOINT`

All orchestration code (`orchestrate.rs`, `lifecycle.rs`) receives a `&dyn ComputeBackend`
and never calls `pg_ctl` directly.

### 10. Project Lease (`tikod/src/lease.rs`)

In a multi-server cluster, at most one server may run a project at a time. The lease is
the distributed fence that enforces this.

**Lease structure** (`{org}/metadata/{proj}/lease.json`)
```json
{ "server_id": "<uuid>", "acquired_at": 1234567890, "expires_at": 1234567950, "generation": 7 }
```

**Acquire lease**
- Attempt a conditional PUT: write only if `lease.json` does not exist or its
  `expires_at` is in the past
- On success: record the lease locally; start the renewal background task
- On conflict: return `LeaseHeld { by: server_id }` — do not start postgres

**Renew lease** (background, every 15 s)
- Update `expires_at = now + TTL` (TTL default 60 s) via unconditional PUT
  (we own it — no CAS needed for renewal)
- If renewal fails (store error): log warning; postgres will be stopped before lease expires

**Release lease**
- Delete `lease.json`
- Called on graceful stop, branch deletion, and migration handoff

**Lease TTL and failure detection**
- A server is considered dead when its leases have all expired (TTL elapsed without renewal)
- Failover loop detects this and re-acquires orphaned leases (see §11)

**Start / stop postgres gate**
- `orchestrate::start` acquires lease before starting postgres
- `orchestrate::stop` releases lease after stopping postgres
- A server must never start postgres for a project without first holding the lease

### 11. Cluster Membership & Failover (`tikod/src/cluster.rs`)

**Server heartbeat** (daemon background task, every 10 s)

Heartbeats are sent via HTTP to a central management server, **not** stored in S3.
tikod exposes a `HeartbeatClient` trait so the concrete transport can be swapped later:

```rust
/// Sends periodic liveness signals and stats to the central management server.
/// Implementation is deferred; the trait isolates the rest of tikod from transport details.
trait HeartbeatClient: Send + Sync {
    /// Report this server's liveness, metadata, and current stats.
    /// Called every 10 s by the heartbeat background task.
    async fn send(&self, payload: &ServerHeartbeat) -> Result<()>;
}

struct ServerHeartbeat {
    // Identity
    server_id: Uuid,
    addr: String,            // e.g. "10.0.1.5:8080"

    // Capacity
    capacity: u32,           // max project slots this server can host
    projects_running: u32,   // currently running projects (held leases)

    // Resource utilisation
    cpu_usage_pct: f32,      // 0–100, averaged over last heartbeat interval
    mem_used_bytes: u64,
    mem_total_bytes: u64,
    disk_used_bytes: u64,    // $PGDATA volume
    disk_total_bytes: u64,

    // I/O stats (delta since last heartbeat)
    io_reads: u64,           // block read ops
    io_writes: u64,          // block write ops
    s3_gets: u64,            // S3 GET requests issued
    s3_puts: u64,            // S3 PUT requests issued

    // WAL archival health (aggregate across locally-held projects)
    wal_archive_lag_bytes: u64,   // unarchived WAL bytes summed across projects
    wal_archive_errors: u32,      // archive_command failures since last heartbeat
}
```

- On daemon start, register with the management server (first `send()` call)
- Refresh every 10 s; log a warning if `send()` fails but keep running
- Stats are collected inside the daemon before each `send()` call; the collection
  logic lives in `cluster.rs` (resource utilisation) and is fed by `wal_health.rs`
  (WAL lag) and `s3_ops` counters (I/O stats)
- **Concrete implementation** (`MgmtApiHeartbeatClient`): POST to
  `{TIKOD_MGMT_URL}/v1/servers/{server_id}/heartbeat` — deferred to future work
- During development / single-server deployments a `NoopHeartbeatClient` is used
  (no-op `send()`, always succeeds)

**Failover loop** (daemon background task, every 30 s)

In the multi-server deployment the management server is authoritative for which servers
are alive. The failover loop will query the management server for the dead-server list.
Until that API exists, failover relies solely on expired project leases (§10) without
needing to know which server died:

1. Scan `{org}/metadata/*/lease.json` across all known orgs
2. If a lease's `expires_at` is in the past and the lease holder is not this server:
   attempt to acquire the lease (CAS on S3 object — fails if another server got there first)
3. On successful acquisition: restart the project here (or forward to a scheduler)
4. Update `ProjectMeta.current_server_id`

This is safe because the lease TTL (60 s) guarantees the previous holder has stopped
postgres before this server takes over.

**Live migration protocol**
```
Source tikod:
  1. compute_backend.checkpoint()         # flush latest state to store
  2. compute_backend.stop()               # stop postgres
  3. lease::release(ns)                   # drop lease
  4. update ProjectMeta.status = "migrating", assigned_server_id = dest

Destination tikod (triggered by HTTP API or failover loop):
  5. lease::acquire(ns)                   # take lease (source has released it)
  6. compute_backend.start(pgdata, env)   # start postgres cold; cache warms on demand
  7. update ProjectMeta.status = "active", current_server_id = self
```
Express cache on the source is abandoned — it is ephemeral and non-authoritative.
